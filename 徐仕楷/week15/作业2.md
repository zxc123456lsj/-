# MinerU vs pdfplumber 文档解析对比

## 一、MinerU 介绍

### 1.1 项目概述

MinerU 是一款高精度文档解析引擎，将 PDF、图片、DOCX、PPTX、XLSX 等复杂文档转换为 Markdown 和 JSON 等机器可读格式，专为 LLM、RAG 和 Agent 工作流设计。项目诞生于 InternLM 预训练过程中，专注于解决科技文献中的符号转换问题。

- **GitHub**: https://github.com/opendatalab/MinerU
- **Star**: 64.4k | **Fork**: 5.4k
- **最新版本**: 3.1.0，升级了 License 至 MinerU Open Source License

### 1.2 核心功能特性

| 功能类别 | 具体能力 |
|---------|---------|
| **支持格式** | PDF、图片、DOCX、PPTX、XLSX |
| **结构处理** | 自动移除页眉、页脚、脚注、页码；保留原文结构（标题、段落、列表）；按人类阅读顺序输出 |
| **内容提取** | 提取图片、图表描述、表格、表格标题、脚注 |
| **公式转换** | 公式转 LaTeX |
| **表格转换** | 表格转 HTML |
| **OCR 能力** | 自动检测扫描件和乱码 PDF；支持 109 种语言 OCR |
| **输出格式** | 多模态 Markdown、NLP Markdown、按阅读顺序排序的 JSON、丰富中间格式 |
| **可视化** | 支持布局可视化、span 可视化，便于确认输出质量 |
| **部署方式** | 内置 CLI、FastAPI、Gradio WebUI，支持本地编排和多服务部署 |

### 1.3 推理后端对比

| 后端 | 兼容性 | 硬件要求 | 适用场景 | 准确度 |
|------|--------|----------|----------|--------|
| `pipeline` | ✅ Linux/Windows/macOS | 4GB VRAM, 16GB RAM | 快速稳定，CPU/GPU 均可 | 85+ |
| `*-auto-engine` | ❌ 仅 GPU | 8GB VRAM | 高精度需求 | 95+ |
| `*-http-client` | ✅ | OpenAI 兼容服务器 | 调用外部模型服务 | - |
| `hybrid` | ✅ | 8GB VRAM | 高精度低幻觉 | - |
| `vlm` | ✅ | 2GB VRAM | VLM 推理 | - |

### 1.4 安装与使用

```bash
# GPU 环境（推荐）
pip install uv
uv pip install -U "mineru[all]"
mineru -p <input_path> -o <output_path>

# 纯 CPU 环境
mineru -p <input_path> -o <output_path> -b pipeline
```

### 1.5 输出格式

MinerU 输出多种格式：
- **Markdown**：多模态 Markdown 和 NLP Markdown
- **JSON**：按阅读顺序排序的 JSON
- **LaTeX**：公式转换为 LaTeX 格式
- **HTML**：表格转换为 HTML 格式
- **中间格式**：丰富的中间格式便于二次处理

## 二、pdfplumber 介绍

### 2.1 项目概述

pdfplumber 是一个基于 pdfminer 和 pdfminer.six 构建的 Python PDF 解析库，专注于提取 PDF 中的文本和表格数据。

- **GitHub**: https://github.com/jsoma/pdfplumber
- **功能定位**: 轻量级 PDF 文本/表格提取工具

### 2.2 核心功能特性

| 功能类别 | 具体能力 |
|---------|---------|
| **文本提取** | 提取每个字符的详细信息 |
| **表格提取** | 支持多种策略（lines、lines_strict、text、explicit） |
| **图形提取** | 提取矩形、线条、曲线等图形元素 |
| **可视化调试** | 提供可视化调试工具 |
| **输出格式** | CSV 或 JSON（JSON 包含更多元数据） |

### 2.3 使用方法

```python
import pdfplumber

with pdfplumber.open("path/to/file.pdf") as pdf:
    first_page = pdf.pages[0]
    text = first_page.extract_text()
    tables = first_page.extract_tables()
```

### 2.4 输出格式

- **纯文本**: 直接提取 PDF 中的文本内容
- **表格**: 返回 Python 列表（嵌套列表）或 CSV
- **JSON**: 包含字符、位置等元数据
- **图形**: 线条、矩形等坐标信息

## 三、MinerU vs pdfplumber 对比

### 3.1 功能维度对比

| 对比维度 | MinerU | pdfplumber |
|---------|--------|------------|
| **支持格式** | PDF、图片、DOCX、PPTX、XLSX | 仅 PDF |
| **OCR 能力** | 内置 109 种语言 OCR | 无（依赖外部 OCR） |
| **表格处理** | 表格转 HTML/Latex，保留结构 | 表格提取为嵌套列表 |
| **公式处理** | 公式转 LaTeX | 不支持 |
| **图片提取** | 原生支持图片提取和描述 | 不支持 |
| **多语言** | 109 种语言 OCR | 主要针对英文/西文 |
| **输出格式** | Markdown、JSON、LaTeX、HTML | JSON、CSV、纯文本 |
| **模型驱动** | ✅ VLM+LLM 驱动，高精度 | ❌ 规则驱动，依赖 PDF 原始结构 |
| **RAG/Agent 适配** | ✅ 专为 LLM、RAG、Agent 设计 | ❌ 数据提取工具，非 AI 应用导向 |
| **部署方式** | CLI、FastAPI、Gradio、Docker | Python 库 |

### 3.2 准确度与适用场景对比

| 维度 | MinerU | pdfplumber |
|------|--------|------------|
| **复杂文档** | ✅ 优秀（布局分析+VLM 理解） | ⚠️ 一般（依赖 PDF 原始结构） |
| **扫描件/图片 PDF** | ✅ 支持 OCR，效果好 | ❌ 无法处理（除非配合外部 OCR） |
| **多栏布局** | ✅ 按阅读顺序重排 | ⚠️ 按 PDF 物理顺序，可能错乱 |
| **公式密集文档** | ✅ LaTeX 转换 | ❌ 纯文本提取，公式散乱 |
| **表格密集文档** | ✅ HTML 格式保留结构 | ⚠️ 表格提取依赖 PDF 线条检测 |
| **性能/资源** | ⚠️ 需要 GPU（pipeline 模式可用 CPU） | ✅ 轻量，纯 CPU 即可 |

### 3.3 核心差异总结

1. **技术路线不同**：MinerU 是 AI 驱动的文档理解引擎（VLM+LLM）；pdfplumber 是规则驱动的数据提取库
2. **信息损失**：MinerU 通过模型理解减少信息损失；pdfplumber 直接提取原始内容，无法理解语义
3. **端到端体验**：MinerU 开箱即用，输出可直接用于 RAG；pdfplumber 输出需二次处理
4. **资源消耗**：MinerU 需要更多计算资源（GPU 推荐）；pdfplumber 几乎零额外资源

---

## 四、结论与建议

- **选择 MinerU**：当你需要处理复杂 PDF（扫描件、多栏布局、公式、图表混合）、需要高保真解析结果、目标是 LLM/RAG/Agent 工作流
- **选择 pdfplumber**：当你只需要快速提取 PDF 中的文本/表格数据、处理大量简单 PDF、资源有限且不需要 AI 能力

MinerU 代表了文档解析的 SOTA 水平（OmniDocBench 86.2 分），pdfplumber 则是轻量级场景下的实用选择。
