# MinerU2.5 论文与使用总结

## 1. 论文对象

本作业阅读的论文是：

```text
Week15/模型论文/2509-MinerU2.5.pdf
```

论文标题为：

> MinerU2.5: A Decoupled Vision-Language Model for Efficient High-Resolution Document Parsing

它介绍了一个面向高分辨率文档解析的视觉语言模型。论文中提到 MinerU2.5 是一个约 1.2B 参数规模的文档解析 VLM，重点目标是在保持较高识别准确率的同时提升计算效率。

## 2. 核心思想

MinerU2.5 的核心是“解耦”和“两阶段解析”。

它不是直接把整张高分辨率 PDF 页面一次性送进模型，而是拆成两个阶段：

```text
阶段一：粗粒度全局版面分析
阶段二：细粒度局部内容识别
```

### 阶段一：Layout Analysis

第一阶段会把文档页面缩放成较低分辨率的缩略图，用于快速识别整体版面结构，例如：

- 标题
- 正文块
- 表格
- 图片
- 公式
- 图注

这样做可以降低直接处理高分辨率图片带来的计算开销。

### 阶段二：Content Recognition

第二阶段根据第一阶段识别出的版面区域，从原始高分辨率页面中裁剪局部区域，再做细粒度识别。

这样可以保留文字、公式、表格等细节，不会因为整页缩放而损失精度。

## 3. 为什么这种设计重要

文档图像和普通图片不同，它通常有：

- 分辨率高
- 文字密度大
- 表格和公式复杂
- 多栏排版
- 阅读顺序不直观

如果直接用通用多模态模型处理整页高分辨率文档，会遇到视觉 token 数量过多、推理成本高、细节丢失或幻觉等问题。

MinerU2.5 通过“先看全局，再看局部”的方式，把版面结构和内容识别拆开处理。用 Java 后端类比，它有点像先用一个轻量服务做路由和定位，再把具体子任务交给专门服务处理。

## 4. 课程中 MinerU 的使用方式

`02_MinerU.md` 中介绍了 MinerU 的实际用途：

- 将复杂 PDF 转换为 Markdown。
- 删除页眉、页脚、页码等干扰信息。
- 按人类阅读顺序输出文本。
- 保留标题、段落、列表等结构。
- 提取图像、表格、公式。
- 自动识别扫描版 PDF 并启用 OCR。

命令行使用方式：

```powershell
mineru -p <input_path> -o <output_path>
```

也可以部署成服务：

```powershell
mineru-api --host 0.0.0.0 --port 8000
```

## 5. 适用场景

MinerU 更适合：

- 复杂 PDF 解析
- 多栏论文解析
- 含图表和公式的技术文档解析
- 扫描件 OCR
- 文档进入 RAG 知识库前的预处理
- 需要 Markdown / JSON 结构化输出的场景

不太适合：

- 只抽取几页纯文本的轻量任务
- 不允许安装较重依赖的环境
- 对响应时间极敏感的在线同步接口

## 6. 和普通 PDF 文本抽取工具的区别

`pdfplumber` 更像一个轻量 PDF 文本/表格抽取库，适合快速读取 PDF 中已有的文本层。

MinerU 更像一个文档解析系统，目标不是只拿到文字，而是尽量恢复文档结构、阅读顺序、图片、表格和公式。

所以在 RAG 场景里：

- 简单 PDF：`pdfplumber` 可能足够。
- 复杂论文、报告、扫描件：MinerU 更有优势。
