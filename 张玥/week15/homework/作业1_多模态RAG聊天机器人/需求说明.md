# 作业 1：多模态 RAG 聊天机器人需求说明

## 1. 背景

课程项目 `05-multimodal-rag-chatbot` 的目标是构建一个能处理图文混排 PDF 的多模态 RAG 聊天机器人。

传统 RAG 通常只处理纯文本，但真实企业文档经常包含：

- PDF 报告
- 图表
- 截图
- 表格
- 图片说明
- 多栏排版

因此，多模态 RAG 不只是“把 PDF 里的文字抽出来”，还要尽量保留图文关系，让模型回答时能引用相关文字和图片。

## 2. 参考项目中的核心需求

参考 `05-multimodal-rag-chatbot/README.md`，原项目希望完成：

- 文档上传：用户上传 PDF、docx、txt 等文件。
- 文档解析：离线调用 MinerU / DeepSeek-OCR 等工具，把 PDF 解析成 Markdown 和图片资源。
- 内容存储：保存原始文件、解析后的 Markdown、图片路径和元信息。
- 向量检索：对文本 chunk 和图片内容建立向量索引。
- 图文问答：用户提问后，检索相关文本和图片，再交给多模态模型回答。
- 来源说明：回答中尽量说明来自哪个文件、哪段资料或哪张图。

## 3. 本作业实现范围

本作业只要求完成“架构和初步代码”，所以采用轻量本地实现：

- 使用本地文件夹保存上传文件和解析结果。
- 使用 SQLite 保存文件元信息。
- 使用 TF-IDF 做本地文本检索，替代云 Milvus。
- 解析 Markdown / txt 文件时直接复制或转换；解析 PDF 时使用轻量文本抽取作为兜底。
- 图片以 Markdown 图片语法保留，例如 `![图1](images/demo.png)`。
- 问答默认生成基于检索片段的模板回答；如果配置了 `OPENAI_API_KEY`，可扩展为真实 LLM 回答。

## 4. 本作业不做的内容

为了避免把作业做成复杂生产项目，本版暂不实现：

- 真实 Kafka 消息队列
- 真实 Milvus / Zilliz 云向量库
- 真实 CLIP 图片向量检索
- 用户权限系统
- 多知识库管理
- 大规模并发处理
- 生产级任务重试和监控

这些能力会在架构文档中说明，但不作为初版代码目标。

## 5. 接口语义

本作业保留原项目的接口语义：

```text
POST /upload/document
  上传文档，记录文件状态为 uploaded

POST /worker/parse/{document_id}
  模拟离线 worker，解析文档并更新状态为 parsed

POST /index/build
  扫描已解析文档，构建本地检索索引

POST /chat
  根据用户问题检索相关资料并生成回答
```

代码中同时提供可直接调用的 Python 函数：

- `upload_document`
- `parse_document_task`
- `build_index`
- `chat_with_rag`
- `render_markdown_with_images`

## 6. 成功标准

作业完成后应能做到：

- 本地 Demo 可以上传示例 Markdown 文档。
- 可以将文档标记为已解析并生成可检索内容。
- 可以根据问题检索相关 chunk。
- 回答中能保留图片 Markdown 链接。
- 代码中不出现硬编码 API Key、Milvus token、Kafka 地址。
