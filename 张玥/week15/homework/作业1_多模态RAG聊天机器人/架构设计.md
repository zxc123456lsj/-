# 多模态 RAG 聊天机器人架构设计

## 1. 原项目架构理解

课程项目 `05-multimodal-rag-chatbot` 中包含几个关键文件：

- `web_page_upload.py`：负责文件上传、写入数据库、向 Kafka 投递待解析任务。
- `offline_precess_worker.py`：负责消费 Kafka 任务，调用 MinerU 解析文件，切分 chunk，并写入 Milvus。
- `web_page_chat.py`：负责用户聊天、问题 embedding、向量检索、图文回答展示。
- `orm_model.py`：使用 SQLite 记录文件元信息。

原始链路可以理解成：

```text
Streamlit 上传页面
  -> 保存文件
  -> SQLite 记录文件
  -> Kafka 投递解析任务
  -> Worker 调用 MinerU
  -> Markdown / 图片产物
  -> BGE / CLIP 编码
  -> Milvus 存储向量
  -> Chat 页面检索
  -> Qwen 模型回答
```

## 2. 作业版轻量架构

作业版保留接口语义，但把复杂外部依赖换成本地轻量组件：

```text
FastAPI / 本地 Demo
  -> upload_document
  -> SQLite 文件元信息
  -> parse_document_task
  -> processed Markdown
  -> build_index(TF-IDF)
  -> chat_with_rag
  -> Markdown 图文回答
```

这样做的好处：

- 不依赖 Kafka、Milvus、云 token。
- 可以在普通本地环境直接运行。
- 便于看清多模态 RAG 的核心流程。
- 后续可以逐步替换为真实生产组件。

## 3. 模块职责

### 文档上传模块

职责：

- 接收文件。
- 保存到 `storage/uploads`。
- 在 SQLite 中记录文件名、路径和状态。

状态：

- `uploaded`：已上传，待解析。
- `parsed`：已解析，已有 Markdown。
- `indexed`：已进入检索索引。

### 文档解析模块

职责：

- 读取上传文件。
- 如果是 Markdown 或 txt，直接转换成标准 Markdown。
- 如果是 PDF，优先保留 MinerU 接入点；本地作业版使用轻量文本抽取兜底。
- 将结果保存到 `storage/processed/{document_id}/content.md`。

生产版本可以替换为：

```text
MinerU CLI / MinerU API / DeepSeek-OCR / HunyuanOCR
```

### 检索索引模块

职责：

- 读取已解析 Markdown。
- 按段落切分 chunk。
- 抽取 Markdown 图片链接。
- 使用 TF-IDF 构建本地检索索引。

生产版本可以替换为：

```text
BGE 文本向量 + CLIP 图片向量 + Milvus
```

### 图文问答模块

职责：

- 接收用户问题。
- 检索相关 chunk。
- 拼接上下文。
- 生成回答。
- 保留 Markdown 图片语法，便于前端渲染。

生产版本可以替换为：

```text
Qwen-VL / GPT-4o / HunyuanOCR VQA / 其他多模态模型
```

### 图片渲染模块

职责：

- 识别回答中的 `![alt](url)`。
- 将文本和图片拆分成结构化片段。
- 前端可根据片段类型分别渲染文本和图片。

## 4. 为什么作业版不直接接 Kafka 和 Milvus

Kafka 和 Milvus 是生产架构中的合理选择，但对作业初版来说成本较高：

- Kafka 需要额外服务和 topic 管理。
- Milvus/Zilliz 需要向量库服务、collection schema 和 token。
- 真实 CLIP 图片检索还需要模型加载和图片向量入库。

本作业重点是先把“上传 -> 解析 -> 检索 -> 问答”的链路讲清楚，后续再把本地组件替换成生产组件。

## 5. 后续可扩展方向

- 把 TF-IDF 替换为 BGE embedding。
- 把本地索引替换为 Milvus。
- 对图片使用 CLIP embedding。
- 文档解析接入 MinerU。
- 聊天回答接入 Qwen-VL。
- 增加知识库 ID 和用户权限。
- 增加任务队列、失败重试和状态监控。
